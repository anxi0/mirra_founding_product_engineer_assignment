# B-context — Part B 작성 배경 대화 정리

## 방향 선택 논의

### 후보 비교

| 방향 | 레버리지 | 리스크 | PoC 범위 |
|------|---------|--------|---------|
| 가 · 부채 | 낮음 (방어적) | 낮음 | i18n 린트, 진입 게이트 UI |
| 나 · 신규 | 높음 (전략적) | 높음 (Agent 범위 큼) | 자연어 → 발행 데모 |
| 다 · 플라이휠 | 중간 (기반) | 낮음 (범위 명확) | end-to-end 미니 파이프라인 |

### 나(Agent-first)를 선택하지 않은 이유

- Agent가 "그냥 말하면 되는" 인터페이스가 되려면 성과 데이터가 내재화되어 있어야 함
- 지금 데이터 파이프라인 없이 Agent만 얹으면 "맥락 없이 답하는 챗봇"이 됨
- 데이터 파이프라인(다)이 Agent-first(나)의 연료

### 가(부채)를 선택하지 않은 이유

- 방어적 작업 — 성장 레버리지가 낮음
- 어떤 부채가 실제 사용자 이탈과 연결되는지 데이터 없이 전면 리팩토링하는 것은 순서가 틀림
- 다가 만든 데이터 루프 위에서 우선순위를 판단하며 점진적으로 해결 가능

### 다(플라이휠)를 선택한 이유

1. **나보다 먼저여야 한다** — `content_performance_map`이 Agent 제안의 근거 데이터
2. **순서 논리가 B의 핵심 논거가 된다** — 나를 선택하지 않은 이유를 설명하면서 나에 대한 이해도를 보여줌
3. **범위가 명확하다** — 스키마 마이그레이션 + 파이프라인 + 분석 루프, PoC로 end-to-end를 작게 돌릴 수 있음
4. **PO 가능한 개발자를 드러내기 좋다** — 데이터 스키마 설계 + 비즈니스 임팩트 연결을 동시에 보여주는 영역
5. **가의 우선순위 판단 도구가 된다** — "어떤 부채가 이탈과 연결되는가"를 데이터로 볼 수 있게 됨

### 다를 선택했을 때의 반론 (인지하고 있음)

- 나를 선택하면 "자연어 → 발행 데모"라는 눈에 띄는 임팩트를 낼 수 있음
- 채용 측 입장에서 동작하는 Agent 데모가 더 강렬할 수 있음
- 단, 과제 평가 기준은 "PO 가능한 개발자"인지 — 순서와 논리를 판단하는 능력이 핵심

## Day별 계획 근거

- **Day 1 현황 파악**: 역공학 스키마와 실제 코드가 일치하는지 먼저 검증. 기준선 수치 없이 1주 후 변화를 측정할 수 없음
- **Day 2 스키마 마이그레이션**: 백필 포함. 기존 데이터를 살리지 않으면 분석 루프에 쓸 데이터가 없음
- **Day 3 이벤트 파이프라인**: PostHog 병합 가능 여부 확인이 핵심. 안 되면 별도 이벤트 테이블 설계로 전환
- **Day 4 분석 루프**: 루프 1·2를 AI 인사이트 API에 연결 → 기존 정적 인사이트를 실데이터로 대체
- **Day 5 검증**: 데이터 정합성 + 쿼리 실행 계획 + 인사이트 1개 이상 도출. C의 PoC 범위 결정

## 1주 후 지표 설정 근거

- **기획→발행 연결률**: 현재 얼마나 끊겨 있는지 보여주는 지표. 높을수록 분석 루프 품질 하락
- **성과 데이터 커버리지**: `post_analytics`가 없는 발행 게시물 비율. 100%에 가까울수록 분석 신뢰도 상승
- **분석 불가 콘텐츠 비율**: `content_idea_id IS NULL` 비율. 기획 없이 만든 콘텐츠는 플라이휠 루프에서 이탈
- **PostHog 병합 가능 여부**: Day 1에 모르고 Day 3에 알게 되는 사항. 결과에 따라 Day 3 계획 분기
