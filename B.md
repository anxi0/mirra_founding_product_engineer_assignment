## 선택: 가 · 부채 (기존 제품 리팩토링)

## 1. 왜 이 방향인가

### 지금 Mirra에 필요한 것은 기능이 아니라 전환이다

Mirra는 기능이 부족한 제품이 아닙니다. SNS 마케팅에 필요한 기획·제작·예약·성과·댓글·DM 기능이 이미 갖춰져 있습니다. 그런데도 수익이 기대치에 못 미친다면, 문제는 기능의 수가 아니라 **사용자가 가치를 경험하기 전에 이탈하는 퍼널**입니다.

기능이 많은 제품이 전환이 낮은 이유는 하나입니다. 사용자가 그 기능에 도달하지 못하기 때문입니다.

### 나(Agent-first)를 선택하지 않은 이유

나는 방향성이 맞습니다. 그러나 지금은 시점이 틀립니다.

Agent-first 인터페이스는 사용자가 "뭘 해야 하는지 알고 있고, 그걸 더 빠르게 하고 싶을 때" 빛납니다. 현재 Mirra는 사용자가 메뉴 구조 자체에서 혼란을 겪는 단계입니다. IA가 기능 단위로 쌓여 있고, 진입 상태 감지가 없으며, 엣지케이스에서 흐름이 끊깁니다. 이 위에 자연어 인터페이스를 얹으면 "말하면 되는" 경험이 아니라 "말해도 안 되는" 경험이 됩니다. Agent는 탄탄한 기반 위에서 가속 도구가 됩니다.

구조적인 제약도 있습니다. Agentic interface는 비용과 신뢰성 문제로 인해 현실적으로 HITL(Human-in-the-loop) 구조로 설계할 수밖에 없습니다. Agent가 생성한 결과를 사람이 검토·승인하는 단계가 반드시 필요하고, 이를 자연스러운 UX로 구현하는 것 자체가 독립적인 설계 과제입니다. 또한 Agent가 실제로 유용한 제안을 하려면 사용자별 콘텐츠 성과 이력, 선호 포맷, 계정 특성 등 방대한 데이터 위에서 정교한 학습이 선행되어야 합니다. 퍼널도 정비되지 않은 상태에서 그 데이터가 쌓일 수 없고, 데이터 없이 Agent를 도입하면 "AI가 붙어 있지만 도움이 안 되는" 제품이 됩니다.

더 근본적인 문제도 있습니다. Agentic interface는 기존 Mirra 사용자층과 사용자 유형이 다를 수밖에 없습니다. Mirra의 현재 사용자가 어떤 페르소나인지 데이터로 확인하지 못했지만, SNS 마케팅 도구의 일반적인 타깃인 SMB 마케터라면 GUI 기반 워크플로우에 더 익숙할 가능성이 높다고 생각합니다. 반면 자연어 인터페이스를 편하게 쓰는 사용자는 AI 네이티브 도구에 이미 노출된 다른 페르소나입니다. 지금 Agentic interface를 도입하면 기존 사용자의 학습 부담을 늘리면서, 새로운 페르소나는 아직 충분히 유입되지 않은 상태가 됩니다. 제품이 두 페르소나 사이 어딘가에서 표류할 가능성이 있습니다.

### 다(플라이휠)를 선택하지 않은 이유

다도 중요한 방향입니다. 그러나 데이터 분석 루프는 **신뢰할 수 있는 퍼널이 먼저 있어야** 의미가 있습니다.

현재 사용자가 기획 페이지에서 이탈하는 이유가 IA 혼란인지, 에러인지, 크레딧 부족인지 지금은 알 수 없습니다. 퍼널이 여러 곳에서 새고 있는 상태에서 쌓인 데이터는 "어디서 새는가"가 아니라 "새고 있다"는 사실만 알려줍니다. 가를 통해 퍼널을 정비하고 이벤트를 심으면, 다의 분석 루프가 비로소 정확한 데이터를 생산합니다. 다 -> 나 순으로 설계 했을 때, 플라이휠을 재 작업해야하는 문제가 발생할 수 있습니다.

### 가를 선택한 이유

Part A에서 발견한 세 가지 구조적 부채는 모두 퍼널 이탈 지점입니다.

**PRD 엣지케이스 공백**은 크레딧 부족·토큰 만료·페이월 중첩 상황에서 흐름을 막습니다. 유료 전환을 앞둔 사용자가 이 시점에 에러를 만나면 이탈합니다. 코드베이스에서 이 세 가지 상태에 대한 진입 전 가드가 없는 것을 직접 확인했습니다.

**IA 흐름 충돌**은 첫 사용자가 "뭘 먼저 해야 하는지" 모르게 만듭니다. Mirra의 실제 이탈률은 확인하지 못했지만, 계정 연동 → 첫 콘텐츠 발행까지의 경로가 사이드바 구조 상 직관적이지 않다고 생각합니다. Day 1에 퍼널 데이터를 열어보는 것이 첫 번째 과제입니다.

**다국어 UI 깨짐**은 글로벌 SaaS에서 신뢰 문제입니다. i18n 키가 그대로 노출되는 화면을 본 사용자는 제품의 완성도를 의심합니다. Mirra는 `/ko/` 경로로 한국어를 지원하고 있으며, 다국어 확장을 전제한 구조이기 때문에 이 문제가 방치되면 해외 진출 시 더 큰 부채가 됩니다.

세 가지 모두 지금 당장 수익 전환에 영향을 주고 있습니다. 그리고 세 가지 모두, 기능을 추가하는 것이 아니라 **기존 코드의 구조를 바로잡는 것**으로 해결됩니다.

지금 Mirra에 가장 레버리지가 큰 한 수는 "새는 퍼널을 막는 것"입니다. 더 많은 기능을 추가하기 전에, 있는 기능이 끝까지 작동하도록 만드는 것이 먼저입니다.

## 2. 첫 1주 계획

스택 기준: Next.js 15 · Vercel serverless · Drizzle ORM · 다국어 SaaS

### Day 1 — 퍼널 지도 작성

- PostHog 접근 권한 확보, 현재 이벤트 목록 파악
- 가입 → 계정 연동 → 첫 콘텐츠 기획 → 첫 발행까지의 퍼널 각 단계별 이탈률 측정
- 코드베이스에서 엣지케이스 3개(페이월 중첩, 토큰 만료, 크레딧 부족)의 현재 처리 방식 확인
- **기준선 고정:** 퍼널 각 단계 이탈률 수치, i18n 키 누락 건수, console error 발생 페이지 목록

### Day 2 — 엣지케이스 가드레일

- 진입 전 가드 레이어 구현: 계정 미연동 / 크레딧 부족 / 토큰 만료 상태를 기능 내부가 아닌 진입 시점에서 감지
- Vitest로 엣지케이스 시나리오를 테스트로 먼저 고정: "토큰 만료 상태에서 발행 시도 시 어떤 상태를 반환해야 하는가"
- 테스트 통과 후 구현 — PRD 공백을 테스트 파일이 채우는 구조

### Day 3 — PostHog 이벤트 계측 강화

- Mirra는 이미 PostHog를 사용 중 — 신규 도구 도입 없이 기존 인프라 위에서 이벤트를 추가한다
- 공통 이벤트 스키마 정의: `event_name`, `workspace_id`, `platform`, `step` 등 공통 프로퍼티 표준화
- 사이드바 메뉴 클릭 이벤트 심기: 어느 경로로 실제 콘텐츠 발행까지 도달하는지 PostHog에 기록
- 기획 → 제작 → 예약 각 단계 진입·이탈 이벤트 추가, PostHog Funnel로 시각화 확인
- IA 재정렬은 이 데이터가 쌓인 후 판단 — 지금은 기존 PostHog에 측정 포인트를 추가하는 것이 먼저

### Day 4 — i18n 가드레일

- 빌드 시점 i18n 키 린트 룰 추가: 번역 키 누락 시 빌드 실패
- 한/영 스냅샷 테스트 설정 (Storybook + Chromatic): CI에서 언어별 UI 자동 비교
- 공통 컴포넌트에 `min-width: 0`, `overflow-wrap`, `line-clamp` 패턴 가이드 적용

### Day 5 — 검증과 정리

- Day 1 기준선 대비 변화 측정: 가드레일 적용 후 에러 발생 시나리오 수, i18n 키 누락 건수
- Vitest 커버리지 확인: 엣지케이스 시나리오 커버리지 목표치 달성 여부
- 1주 성과 문서화 및 다음 우선순위 판단 — Day 3에 심은 이벤트 데이터를 보고 IA 재정렬 범위 결정

### 1주 후 달라진 상태

지금은 사용자가 기능 내부에서 에러를 만나고 이탈합니다. 1주 후에는 문제 상황을 진입 전에 감지하고 해결 경로를 제시합니다. 퍼널 이탈이 어느 단계에서 발생하는지 데이터로 보이기 시작합니다. 이 데이터가 다음 스프린트의 IA 재정렬 범위와 Agent-first 전환 우선순위를 결정하는 근거가 됩니다.
